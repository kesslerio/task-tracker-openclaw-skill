# v0.3.1 Evidence Schema

This contract defines the U1 evidence record, stable standup window, and
idempotency rules used by later source adapters.

## Evidence Record

Adapters emit canonical dictionaries through `scripts/evidence_record.py` with
`schema_version: 1`. The required fields are:

- `source`: `github`, `gmail`, `calendar`, or `dialpad_sms`.
- `kind`: `activity` or `commitment` for adapters. `accomplishment` is reserved
  for the confirm gate only.
- `provider_id`: the stable provider identity, such as `repo#123`, `repo@sha`,
  a Gmail message/thread id, a calendar event occurrence id, or a Dialpad
  message id.
- `provider_state`: the provider's update/status token. It is part of the
  reprocess decision.
- `evidence_hash`: `harvest_ledger._evidence_hash(source, provider_id)`.
- `occurred_at`: ISO timestamp in the local Pacific timezone. The evidence
  window is based on this timestamp, not ingest time.
- `match_title`: clean fuzzy-match text. Display refs such as `[repo#123]` stay
  out of this field.
- `title`: human display text.
- `url`: provider URL or `null`.
- `match`: matcher metadata or `null`.
- `auto_done_eligible`: always `false` for `calendar` and `dialpad_sms`.
- `run_id`: optional immutable id for the invocation that produced the record.

Calendar and SMS records are evidence candidates only. They never become DONEs
without the confirm gate.

## Reconciliation Into Standup Completed

`scripts/reconcile.py` is the only U5 path that combines user-stated completions
with harvested evidence for `output["completed"]`.

The invariant is:

- `output["completed"]` contains only confirmed accomplishments: daily-note
  DONES, board `[x]` items, and future confirm-gate approved items.
- Raw harvested evidence never creates a completed item, even when the matcher
  found an explicit task reference. Unmatched evidence remains in
  `output["evidence_candidates"]` for confirmation.
- Calendar and `dialpad_sms` records remain evidence candidates unless they
  match an already confirmed user claim. Their `auto_done_eligible=false` flag is
  preserved in provenance and is never used as a promotion bypass.
- When evidence matches a confirmed user claim by `evidence_hash`,
  `matched_task_id`, or normalized-title equality, the completed item keeps the
  user's title and fields. Evidence is attached under `provenance`; the evidence
  title never overwrites the user wording.
- A candidate credited as provenance for a confirmed completed item is removed
  from `output["evidence_candidates"]` so it is not shown twice.
- Reconciliation is idempotent: rerunning over the same completed/provenance set
  does not duplicate rows or provenance entries.

Evidence provenance entries carry the source plus provider identity where
available: `source`, `kind`, `title`, `url`, `evidence_hash`, `provider_id`,
`matched_task_id`, and `auto_done_eligible`. User-claim provenance can be added
when a claim is merged with evidence or deduped with another claim; it records
the user source without minting `kind="accomplishment"`.

### `source` vs legacy `source_type` (a U2 integration constraint)

The canonical record stores the provider in BOTH `source` and `source_type`
using the **provider taxonomy** (`github` / `gmail` / `calendar` / `dialpad_sms`),
and `evidence_hash` / `match_evidence` key off that taxonomy consistently within
the standup path. This is **distinct** from the legacy weekly-ledger taxonomy that
`harvest_ledger._classify_match_bucket` / `bucketise` expect (`pr` / `email`).

Therefore a U2+ adapter MUST NOT feed canonical standup records into the weekly
digest's `_classify_match_bucket` / `bucketise` unmapped — a `gmail` record would
misbucket as "Advanced" instead of "Maintenance". The standup path has its own
`evidence_candidates` rendering; if any unit ever needs the weekly bucketiser, it
must first map the provider taxonomy to the legacy `pr` / `email` categories (and
note that `github` now spans PRs **and** commits, and `calendar` / `dialpad_sms`
have no legacy category). In U1 nothing emits canonical records into the weekly
classifier, so this is a forward constraint, not a live defect.

## Standup Window

`scripts/harvest_window.py` resolves two explicit dates:

- `plan_date`: the day `/standup` is planning.
- `evidence_date`: the local Pacific workday being summarized.

Default morning-standup behavior is prior-workday calendar-day evidence:

- A Tuesday 8am Pacific standup has `plan_date=Tuesday` and summarizes Monday
  `00:00 <= occurred_at < Tuesday 00:00` Pacific.
- A Monday standup summarizes the prior Friday.
- Early-today items are excluded from that morning evidence window.

Manual reruns pass `target_date` as the explicit evidence/label date. The same
target date always derives the same plan label, ISO week id, evidence date, and
Pacific boundaries regardless of wall clock. Backfills can additionally pin
`evidence_date` when the summarized day differs from the label date. A run at
00:05 Pacific with no target date uses the new Pacific calendar day as
`plan_date` and the default prior-workday evidence window.

DST is represented by local-midnight boundaries in `America/Los_Angeles`, so
spring-forward windows are 23 UTC hours and fall-back windows are 25 UTC hours.
ISO week ids come from `harvest_state.iso_week_id(plan_date)`, including
week-year boundaries.

## Source Adapter Semantics

Later adapters must query with the U1 window plus a small overlap and then filter
by `occurred_at`.

- `filter_records` passes a record with no `occurred_at` through unchecked. U2
  source adapters must populate `occurred_at`; it is the window-filtering key.
- Overnight/cross-midnight timed items count on the date of their `occurred_at`
  start. Later units may split display, but identity stays stable.
- All-day calendar events are not activity evidence.
- Recurring calendar events use occurrence-level ids.
- Cancelled/declined events are not surfaced as activity, but a changed
  `provider_state` for a previously seen identity is reprocessed so stale
  evidence can be cleared.
- Dialpad SMS uses message ids and deterministic filters; raw bodies are not part
  of this U1 contract.

## Idempotency And Concurrency

Identity is `evidence_hash`; freshness is `(evidence_hash, provider_state)`.
Same identity and same state is skipped on overlap rereads. Same identity with a
changed state is reprocessed.

Standup evidence windows persist to their own `harvest-state-standup.json` file
with sidecar `harvest-state-standup.lock`. This is isolated from the weekly
`harvest-state.json` and the 24h `/done` `harvest-state-24h.json`, mirroring the
existing weekly/24h separation.

The harvest state files store:

- `run_id`: the id stamped on the latest invocation state write.
- `watermarks`: per-source high-water marks.
- `seen_hashes`: legacy identity list.
- `seen_provider_states`: state-aware dedup map.

State writes use an exclusive sidecar flock over the matching lock file and
persist through `utils._atomic_write`. The write path merges with the current
on-disk state so concurrent cron and manual runs do not lose each other's
updates. Across windows, the fresher lexicographic `harvest_window_id` wins so
an older save cannot clobber a newer window. Within one window, lists union,
scalars take the latest non-`None` value, watermarks take the latest instant,
and `seen_provider_states` takes the current run's observation.
