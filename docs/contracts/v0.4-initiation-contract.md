# v0.4-C initiation nudge — decision contract

**Status:** C1 (foundation) shipped — the delivery identity, the expiring proposal, and the
send-time compare-and-swap (CAS). The evaluator (C3), the dispatcher (C4), the point-in-time
calendar read (C2), and the holdout/metrics (C5) extend this file as they land.

The initiation nudge is the v0.4 "Chief-of-Staff pushes you to *start*" lever: when you have
committed a #1 for the day and have **not started it**, after a grace period and only if you are
not visibly busy, the system delivers ONE leashed "▶️ Start it?" nudge. It is a **generalization of
the shipped `checkin_dispatch` pattern** (deterministic COMMAND cron → reload → re-prove target →
inert template → `outbox.deliver_once`), not a new delivery path. This contract pins the data shapes
and the stale-state defense; it deliberately reuses `outbox`, `nag_state`, and the focus session.

## 1. The episode slot (`focus_episode_id`)

A cold-start nudge fires **before any focus session exists** — there is no `nag_state` `session_id`
to key on at decision time. So the episode identity is a deterministic **slot** that exists the
moment a #1 is committed:

```
focus_episode_id = focus_episode_slot(user_scope, task_id, local_date)
                 = "<user_scope>:<task_id>:<local_date>"      e.g. work:tsk_ab…:2026-06-24
```

- Stable for a given (scope, task, day) → the evaluator can run repeatedly and the holdout arm (C5)
  keyed on the slot is stable; identical across the treatment and control arms.
- `user_scope` is carried **inside** the slot — this realises the checkpoint's
  `(user_scope, focus_episode_id, stage)` key tuple without duplicating scope as a separate key
  segment.
- Segments must not contain `":"` (the outbox key is colon-joined) — `focus_episode_slot` raises on a
  stray colon or empty segment rather than mint an ambiguous/colliding key. Real values never contain
  one (`work`; `tsk_…`; ISO `YYYY-MM-DD`).
- `user_scope = "work"` for the entire C-slice; the personal-board lane is deferred to v0.4-A.

When the user finally taps **Start**, the `nag_state` focus session binds to this slot (its `task_id`
matches), so the slot and the eventual session are linked without the slot depending on the session.

## 2. The outbox kind + idempotency key

`outbox._KNOWN_KINDS` gains `"initiation"`. The at-most-once key is:

```
initiation:<focus_episode_id>:<stage>     # make_idem_key("initiation", focus_episode_id, stage)
```

`stage ∈ { cold_start, cold_start_renudge }`. The slot+stage (NOT a clock period, NOT a focus-session
id) is the dedupe identity, exactly as `checkin` keys on `(session_id, phase)` and `nag` on
`(task_id, period)`: a deterministic-dispatcher cron **retry** can never double-send the same stage's
nudge. Delivery route / attempt count stay **out** of the key (they live in the receipt), so a retry
to a re-proven target still dedupes.

## 3. The expiring proposal (`initiation_contract.Proposal`)

The C3 evaluator writes a `Proposal`; the C4 dispatcher reads it. Immutable — a new decision is a new
proposal. Fields:

| field | meaning |
|---|---|
| `focus_episode_id` | the slot (§1) — must equal `focus_episode_slot(user_scope, task_id, local_date)` |
| `task_id` / `user_scope` / `local_date` | the committed-#1 identity (queryable, also encoded in the slot) |
| `stage` | `cold_start` or `cold_start_renudge` |
| `reason_code` | why the evaluator fired (e.g. `committed_unstarted`) — for metrics/audit |
| `created_at` / `expires_at` | ISO UTC; the proposal is dead once `now >= expires_at` |
| `cas_focus_state_rev` | snapshot of `focus_state.rev` at write time (task-state CAS token, §4) |
| `cas_no_session_since` | the instant after which ANY focus-session start/end for the task invalidates (focus-episode CAS token, §4) |
| `arm` | the holdout arm (`treatment`/`control`) — set by C5; `None` here |

`is_expired(now)` treats an unparseable `expires_at` as expired (a proposal we cannot date must not be
acted on). `from_dict` ignores unknown keys (forward-compat as later units add fields).

## 4. The send-time CAS (`cas_still_valid`)

Run by the dispatcher **inside the outbox flock** immediately before delivery (the serialized RMW is
the atomic claim — no new lock). Guards **stale state, NOT injection** (injection is handled at render
time: fixed templates, escaped titles, no raw task/email body). Two dimensions, **both** must hold:

1. **Task-state version** — `focus_state.rev` (a monotonic integer bumped on every `save_focus_state`,
   read via `focus_state.current_rev()`) must equal `cas_focus_state_rev`. A bump means the user
   re-proposed / approved / vetoed — the #1 the nudge is about may have changed → abort. `None`
   (the committed-#1 state is gone/corrupt) → abort. *Why a counter, not the `updated_at` timestamp:*
   a 1-second ISO stamp is coarse and clock-sensitive; the integer is an unambiguous version. It
   never regresses even when the morning re-propose builds a fresh state document — `save_focus_state`
   floors the next value against the value already on disk, and the read-bump-write runs under a
   **sidecar flock** (`focus-state.lock`) so two concurrent writers (the proposer cron vs a user
   `/approve`/`/veto`) can't both mint the same `rev` with different content.
2. **Focus-episode version** — no focus session for the task may have `started_at` **or** `ended_at`
   at/after `cas_no_session_since`. A new start means the user already began (nudging now is a false
   positive); an end means they finished/cancelled. The session lifecycle is inherently monotonic
   (`started_at`/`ended_at` only move forward), so no extra counter is needed here.

`cas_still_valid` is a **pure** function (`current_focus_rev`, `task_sessions`) for exhaustive unit
testing; `cas_still_valid_now` reads live state and **fails CLOSED** on any read error — a send we
cannot prove safe is not sent. A present-but-unparseable session timestamp also fails closed (we
cannot prove the session is older than the baseline). Because `nag_state.read_state()` *quarantines* a
corrupt file to an empty dict — indistinguishable from "no sessions" — `cas_still_valid_now` probes the
raw nag-state file first and fails closed on a present-but-unparseable / non-object one (it does not
read "the user has not started" from a file it could not read).

## 5. The proposal store (`initiation_store`)

`initiation-proposals.json`, keyed by `focus_episode_id` (at most one pending proposal per slot; a
re-nudge supersedes). Read/write/clear under an exclusive sidecar flock + `utils._atomic_write` — the
same idiom as `outbox` / `nag_state.transition`. Expired entries are pruned on every read and write so
the file stays flat. A corrupt file reads as empty (no pending proposal → no send) rather than crashing
the dispatcher. `clear_proposal` is called by the dispatcher once a slot is delivered/abandoned.

## 5b. Known residuals (deferred — from the C1 adversarial review)

- **Silent suppression has no telemetry yet.** Every CAS/store fail-closed path returns "do not send"
  without a log, so a systemic read failure (which suppresses *every* nudge) looks identical to healthy
  "nothing to nudge". **Deferred to C4**, which owns the dispatcher's send-rate / `cos_health` receipt and
  can distinguish `suppressed:read_error` from a legitimate stale-CAS suppression without logging PII.
- **Schema-drift on a rollback.** `initiation_store` prunes any entry the current `Proposal` cannot
  reconstruct (treated as expired). A rollback across a future field rename could therefore drop live
  entries. Mitigation is operational: migrate the store (or let it drain) across a `Proposal` field rename;
  do not roll back code over live proposals written under a newer schema.
- **Backup-restore can lower `focus_state.rev`.** Restoring an older `focus-state.json` over a newer one
  resets `rev` downward. This is safe for the CAS — a proposal snapshot taken at the higher `rev` then
  `!=` the restored value and is **suppressed** (fail-closed), never falsely sent — but operators should
  prefer not to restore focus-state under a live pending proposal.

## 9. The dispatcher — `initiation_dispatch` (C4 — shipped)

The "hands deliver" half — `checkin_dispatch` generalized from body-double check-ins to initiation. A
recurring deterministic COMMAND cron (`telegram-commands.sh initiation-tick`, `sh -c` U8-parity form, NO
LLM turn) runs `run_tick` each tick: it calls the C3 evaluator (`decide_and_store`) to park a proposal if
one is due, then dispatches the pending proposal for today's committed-#1 slot.

- **`run_dispatch(focus_episode_id)`** mirrors the check-in discipline:
  1. **reload** the proposal; absent/expired → send nothing, clean exit.
  2. **send-time CAS** — `cas_still_valid_now` (the #1/priorities `rev` + the focus-episode lifecycle). A
     STALE proposal is **cleared and dropped**.
  3. **snooze + calendar** are *wait* states, not staleness — they **leave** the proposal parked for a
     later tick within its TTL (waiting for a free, un-snoozed moment is the adaptive "good moment").
  4. **re-prove the target NOW** (`nag_delivery.resolve_target`, Hard Gate #4) — transient failure leaves
     it parked.
  5. **render INERT templated text** — a fixed "Start it?" nudge with the committed #1's title sanitized
     to a single capped line (no raw task body → no instruction channel) + the shipped `priority_nag_row`
     (Start / Done / Snooze).
  6. **`outbox.deliver_once`** keyed `initiation:<slot>:<stage>`, passing the CAS as an **in-flock
     `precheck`** — the atomic claim: a proposal that goes stale between the re-check and the held lock
     **aborts** without sending or recording (the slot stays open for a corrected re-fire).
  7. **clear** the proposal once terminal (delivered / already-delivered / aborted-stale).
- **`outbox.deliver_once` gained an optional `precheck`** (additive; every existing caller passes `None`):
  evaluated inside the flock, after the dedup, before the sender — `False` aborts the send. This realizes
  the checkpoint's "recheck + deliver inside the existing outbox flock" without a new lock/scheme.
- **`initiation_cron_descriptor`** is a **code-only** template (`payload.kind="command"`, no
  `delivery.announce` — the dispatcher OWNS the send; no real chat id committed; env-var NAMES only). Live
  registration + the schedule cadence are a **deferred OPERATOR step gated on the C5 holdout existing**.
- Ledger events `initiation_sent` / `initiation_delivery_blocked` (mirroring `checkin_*`); `run_main`
  envelope so any crash is logged + health-recorded and exits 0.

## 8. The state-machine evaluator — `initiation_eval` (C3 — shipped)

The genuinely-new "adaptivity": a **pure, rules-only** evaluator (U9 discipline — no model, fail-OPEN
toward silence, gates by explicit stored facts) that decides whether to emit a `Proposal`. No send, no
board mutation.

- **`evaluate(now, *, user_scope="work") -> Proposal | None`** runs five gates, **cheap-to-expensive** so
  the calendar subprocess only runs once everything else passes:
  1. **committed #1** — `focus_state` is APPROVED for today and `daily_priorities[0]` gives the task;
     `approved_at` is the commit instant the elapsed clock runs from (a stale/merely-proposed list is not
     a commitment, the same rule the nag path uses).
  2. **not started** — no focus session has engaged the task today. "Engaged" = an active session, OR any
     session whose start / scheduled-end (`ends_at`, so an overnight block running into today counts) /
     explicit-end falls on today, OR an ended session we cannot date (fail CLOSED — a garbage stamp on a
     real same-day session must never read as "not started" and fire a false nudge).
  3. **not snoozed** — `nag_state.is_snoozed` on the task's nag entry.
  4. **cadence / budget** — `cold_start` once after `initiation_elapsed_min` (90) past the commit; one
     `cold_start_renudge` after `initiation_renudge_after_min` (120) past the **cold-start send**; both
     capped by `initiation_daily_budget` (2). The two stages' **outbox receipts ARE the send history**
     (`get_receipt` on each stage key) — with one committed #1/day the per-slot count is the per-day
     budget, so no separate counter is needed.
  5. **calendar** — `availability.not_known_busy(now)` (C2) must be clear.
- Emits a `Proposal` with the CAS tokens snapshotted (`focus_state.current_rev()`, `cas_no_session_since =
  now`) and **`arm` unset** (C5 stamps the holdout arm). **Fail-OPEN:** any read error → `None` (no nudge)
  — for a proactive send, a missed nudge is harmless, an errant one is not.
- **`decide_and_store(now)`** is the evaluator's only side effect: `evaluate` + `initiation_store.write_proposal`,
  the durable hand-off to the C4 dispatcher's separate send-time re-check tick. Cadence values are
  `cos_config` knobs (`INITIATION_ELAPSED_MIN` / `_DAILY_BUDGET` / `_RENUDGE_AFTER_MIN` / `_PROPOSAL_TTL_MIN`).

## 7. Point-in-time availability — `availability.not_known_busy(now)` (C2 — shipped)

The U3 `calendar_adapter` is a **window** read (a day of accepted/organized events as standup
evidence). The nudge needs the opposite shape: at the send instant, **"is the user in an accepted
meeting right now?"** — a point-in-time *containment* check.

- **`not_known_busy(now) -> bool`** returns **True** (safe to nudge) ONLY on a *fresh, successful*
  calendar read in which **no accepted/organized timed event contains `now`**. It returns **False**
  (suppress) when an accepted event contains `now`, OR availability cannot be confidently determined.
  The double-negative name is from the seed/checkpoint; the contract is "True = positively confirmed
  free at `now`".
- **Fail CLOSED** on every uncertainty: **no calendar configured**, malformed `STANDUP_CALENDARS`, the
  breaker open, a `gog` non-zero exit / timeout / unparseable output, or an accepted event we cannot
  time → all return False. A nudge fired mid-meeting is the worst false positive, so any doubt is
  silence. *(No-calendar-configured failing closed means a user with no calendar gets no initiation
  nudges — the conservative reading of decisions-C4 "fail closed on unavailable"; flip a config flag
  later if that's not wanted.)*
- **Classification is single-sourced** from `calendar_adapter._event_allowed` (timed · accepted or
  self-organized · NOT cancelled / declined / **all-day**), so "what counts as busy" cannot drift
  between the harvest read and this one. **All-day events do NOT count as busy** (decisions-C4).
- **Event-type policy is availability-specific, NOT the harvest denylist.** A `focusTime` block and an
  `outOfOffice` period ARE busy (they are exactly "do not interrupt me" / "I'm away") — the harvest's
  `_DENIED_EVENT_TYPES` deliberately drops them as *non-evidence*, which is the OPPOSITE of "not busy", so
  reusing it would nudge straight through a focus block. Only `birthday` / `workingLocation` are treated as
  non-blocking (`_NON_BLOCKING_EVENT_TYPES`); everything else is classified by `_event_allowed`.
- **Unparseable gog output fails CLOSED, strictly.** Unlike the harvest's lenient `_events_from_payload`
  (empty harvest evidence is fine), the gate uses `_strict_events`, which RAISES on a non-list `events`
  payload (`{"events": "<error>"}`, `null`, a scalar, a wrapper without events) so it suppresses rather than
  reading the odd shape as "no meetings". The per-calendar read is wrapped in a broad `except Exception` →
  suppress, so an unexpected parse/classify error can never fall through to "free". A reversed `end <= start`
  interval is also treated as untimeable → busy.
- **Residuals (deferred):** (a) a `tentative` RSVP currently classifies as *not busy* (it is neither
  accepted nor organized) — revisit toward busy if mid-tentative-meeting nudges appear; (b) if `gog` ever
  prints the access token into stderr as *free-form text* (not the `--access-token`/`Bearer`/`Authorization`
  forms `error_envelope` redacts), it would persist to the owner-only (`0o600`) degraded log — an exposure
  inherited from the U3 harvest read; the proper fix (token via env/stdin, not argv) spans both reads and is
  a separate hardening.
- **Freshness** is by construction: the read is on-demand at `now` (the dispatcher re-calls it inside
  the send-time recheck), so the decisions-C4 **10-min staleness** bound is satisfied — nothing caches
  an availability reading across minutes. A 12 h look-back window catches a long meeting still running.
- Reused seams: `_calendar_configs` (env `STANDUP_CALENDARS`), the `gog` subprocess shape + token-env,
  and `error_envelope` breaker/`log_degraded` (component `initiation:availability`).
